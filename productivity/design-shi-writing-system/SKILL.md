# 设计虱写作系统 Skill

**适用场景**：管理设计虱自媒体文章的完整创作流程（从灵感到发表）
**作者**：醉雪梅花（设计虱）
**工作目录**：`/opt/data/workspace/设计虱-写作系统/`

---

## 核心理念

**类比知识库**：知识库沉淀"过去经验"，写作系统沉淀"未来内容"。两者并行、互相引用、不合并。

**和知识库的关系**：
- 写作时从知识库拉背景知识
- 好的写作洞察可以沉淀到知识库
- 入口分开，避免混乱

| 目录 | 谁管理 | Agent 行为 |
|------|--------|------------|
| 灵感库/收集箱 | **人** | 只读取，不修改 |
| 灵感库/灵感卡片 | **Agent** | 创建、更新、整理 |
| 选题库 | **Agent** | 创建、更新、移动状态 |
| 稿件库 | **Agent** | 创建、更新 |
| 已发表 | **人** | 人手动放入 |
| 数据分析 | **Agent** | 创建、更新分析报告和数据追踪 |

- 收集箱是素材入口，素材第一时间丢这里
- Agent 只读收集箱，定期整理为灵感卡片
- 已发表由人手动管理，Agent 不碰

## 工作流程

### 1. 素材入口（人）
看到好文章片段、零星灵感、有启发的想法 → 丢进 `灵感库/收集箱/`
- 文件名：`YYYY-MM-DD-HH-MM-简述.md`
- 内容：直接粘贴原文或写下想法，不用整理

### 2. 素材整理（Agent）
定期（每周或有新素材时）整理收集箱：
- 读取收集箱的原始素材
- 创建/更新 `灵感库/灵感卡片/` 下的结构化卡片
- 提炼：核心洞察、可延伸方向、关联性
- 打标签（tags）
- 整理完后可清空或标记已处理的收集箱文件

### 2.5 风格提取（写之前）

**触发条件**：首次写作，或用户说"分析一下我的风格"。

**操作步骤**：
1. 读取 `已发表/` 目录下 3-5 篇最新文章
2. 分析并提炼风格特征，输出 **账号风格档案**
3. 存入 `写作系统/风格档案.md`，后续写作自动引用

**分析维度：**

| 维度 | 提取内容 |
|------|----------|
| 人称视角 | 用"我"还是"我们"？是否直接对读者说话？ |
| 段落节奏 | 平均段落长度？长短句比例？一句话成段的频率？ |
| 句式特征 | 常用转场词？口语化程度？是否有标志性口头禅？ |
| emoji使用 | 频率？位置（段首/段尾/不用）？类型偏好？ |
| 论证方式 | 以个人体验为主？数据为主？类比为主？ |
| 结尾风格 | 收尾方式（行动呼吁/金句/开放式/干脆利落）？ |
| 禁区偏好 | 哪些写法是绝对不碰的？ |

**风格档案格式**（存入 `写作系统/风格档案.md`）：
```markdown
---
type: style_profile
updated: YYYY-MM-DD
source_articles: [文章1, 文章2, 文章3]
---

# 设计虱账号风格档案

## 人称与视角
（分析结果）

## 节奏与句式
（分析结果）

## 语言特征
（口头禅、转场词、情绪表达方式等）

## emoji 使用
（频率、位置、类型）

## 结尾模式
（常用收尾方式）

## 绝对禁区
（从已发表文章反推的禁区）
```

**注意**：
- 风格档案不是死规则，是参考基线
- 写作时如果和"最优写法"冲突，优先保风格一致性
- 每发表 5-10 篇后更新一次风格档案

### 3. 选题提炼（Agent）
从灵感卡片组合中提炼选题：
- 选择角度互补或有关联的卡片
- 创建 `选题库/pending/` 下的选题文件
- 关联来源卡片（inspiration_sources）
- 规划写作大纲

### 4. 写作（Agent）
接到写作任务时：
- 确认选题状态改为 writing
- 从选题关联的灵感卡片中提取素材
- 必要时检索知识库补充背景
- 按 design-shi-article skill 的规范写作
- 输出到 `稿件库/`

### 4. 用户修改（人）
Agent 写完原稿后：
- 等待用户手动修改、调整、配图
- **不要自动创建五平台分化版本**，等用户明确说"分发"或"适配各平台"时再做
- 用户确认终稿后，放入 `已发表/`

### 4.1 微信公众号摘要规范
**触发条件**：需要为公众号版本添加摘要

**操作步骤**：
1. 在原文末尾「---」后添加引用块格式的摘要
2. 内容 ≤ 100 字
3. 用 humanizer-zh 改写，使语言更自然、像人手写
4. 示例：
```
> 摘要：在这篇文章里，我用一次真实的故障排查经历，带你理解了 AI 助手'失忆'的真正原因——不是上下文污染，而是推理模型无法做摘要导致的系统强制重置。看完这篇，下次遇到 AI 突然失忆，你就知道该检查什么了。
```

### 5. 五平台分发（Agent）
**触发条件**：用户确认"分发"

**操作步骤**：
1. **读取原稿**，确认文章内容
2. **从原稿末尾的「发布建议」读取5个平台标题**（不是用之前文章的标题！逐字复制标题内容）
3. 在 `稿件库/已适配/` 下创建文件夹，将原稿移入
4. 创建5个平台版本，标题按「发布建议」配置
5. **保留原稿中的所有图片**，在对应位置插入到各平台版本
6. **生成5张封面**，标题从「发布建议」读取

**关键注意事项**：
- ⚠️ 标题必须从**当前文章**的「发布建议」读取，**逐字复制**，不要改动、不要参考之前文章
- ⚠️ 分发版本必须保留原文图片，按原文位置插入（用关键词定位插入点）

### 6. 发表归档（人）
用户手动：
- 将终稿放入 `已发表/`
- 更新选题状态为 completed
- 填写产出文章链接

### 6.5 发表后归档（2026-05 确认）
**触发条件**：用户告知某篇文章已发表

**操作步骤**：
1. 在 `稿件库/已适配/` 找到对应分发文件夹
2. **只保留原稿 `[原稿]xxx.md`**（删除所有5平台分发版、封面、docx等）
3. 原稿文件名前加日期前缀（如 `2026-05-21-文章标题.md`），直接移动到 `已发表/YYYY/` 目录（**不用套文件夹**）

**注意**：分发文件夹移动后即为已发表归档，不需额外压缩

## 五平台适配流程

**触发条件**：稿件完成后，用户要求适配多平台发布

**操作步骤**：
1. 读取稿件库中的原稿
2. 在 `稿件库/已适配/` 下创建 5 个版本，命名格式：`YYYY-MM-DD-平台-文章标题.md`
3. 每个版本替换对应平台标题（原稿顶部有五个标题）
4. 按平台调整内容深度

**各平台适配规则**：

| 平台 | 内容深度 | 标题风格 | 特殊处理 |
|------|---------|---------|---------|
| 头条号 | 完整版 | 数字+实用推荐 | NAS/数码话题发，AI话题不发 |
| 什么值得买 | 完整版 | [标签]+实用 | 突出性价比/攻略 |
| 知乎 | 完整版 | 问题式+洞察 | 突出技术分析 |
| 微信公众号 | 精简版（~40%内容） | 痛点式（≤20字） | 前言尾注用反引号包裹 |
| 小红书 | 大幅精简（~20%内容） | emoji+感叹号 | 重新做封面，加活动标签 |

**排版规范**：
- 小标题格式：`#### | 标题`（不加颜色 span，微信公众号用编辑器上色）
- Docker 部署统一用 compose 格式（绿联 NAS 有 Docker Compose 管理器，粘贴即部署，不写 docker run 和 docker compose up 命令）
- 前言/尾注：反引号包裹，**不带** `|`

**小红书特殊处理**：
- 检查是否有活动话题可参与（如 #榨干设备howto）
- 标题加活动格式前缀（如 `howto|`）
- 末尾加 `@活动官号 #活动标签`
- 封面图：写完小红书文章后**自动生成** HTML 封面（见下方流程）

### 小红书封面自动生成流程

**触发条件**：小红书适配版文章写完后自动执行，无需用户确认。

**操作步骤**：
1. 根据文章核心卖点，设计封面内容（标题 + 2-3 个功能亮点）
2. 生成 1080×1440（3:4 竖版）HTML 文件，保存到 `/tmp/img-output/xhs-cover.html`
3. 用 `browser_navigate` 加载 HTML，`browser_vision` 截图
4. 将截图复制到 `写作系统/配图/YYYY-MM-DD-文章简称-小红书封面.png`
5. 告知用户封面已生成，附截图路径

**HTML 封面模板要点**：
- 字体：`'Noto Sans CJK SC', 'WenQuanYi Micro Hei', sans-serif`（开源免费商用字体）
- 已弃用 PingFang SC、Microsoft YaHei、Hiragino Sans GB（版权风险）
- 中文字体依赖：已安装 `fonts-noto-cjk` + `fonts-wqy-microhei`

**统一视觉风格（v2.0 定稿 2026-05-20）**：
所有封面共用同一套视觉语言，不再按话题换配色。

- **竖版**（小红书 1080×1440）：
  - 模板：`/opt/data/workspace/Projects/html-ppt-skill/templates/cover-templates/vertical-1080x1440.html`
  - 背景：深紫蓝渐变（#0c0c1d → #1a1a3e）+ 4 个 mesh gradient orbs（粉红 #ff6b6b、青 #4ecdc4、紫 #a29bfe）
  - 叠加：SVG feTurbulence 噪点 + 2 条装饰竖线
  - 结构：标签（左上角胶囊）→ 3 个连接圆点 → 大标题（渐变文字）→ 副标题 → 蓝紫渐变横杠 → 毛玻璃卡片（3 个 feature：图标+标题+描述）→ 底部署名
  - 毛玻璃：`rgba(255,255,255,0.07)` + `backdrop-filter:blur(20px)` + 1px 半透明边框 + 28px 圆角
  - **html 必须加 `html { background: #0c0c1d; }`** 防止浏览器视口大于 body 时出现边框色差

- **横版大尺寸**（公众号/什么值得买 1550×660）：
  - 模板：`/opt/data/workspace/Projects/html-ppt-skill/templates/cover-templates/horizontal-1550x660.html`
  - 背景：深紫蓝渐变（#0e0e28 多层 radial-gradient）+ 5 个 mesh gradient orbs（粉红、青绿、紫色）
  - 叠加：36px 网格线（rgba(78,205,196,0.07)）+ SVG 噪点
  - 结构：外层 `.glass` 毛玻璃大卡片占满全屏（留 18px 边距）→ 左侧：tag-row → 标题（两行：白+灰）→ 描述 → 作者；右侧：核心优势卡片（3 个 dot+文字条目）
  - 毛玻璃：`rgba(255,255,255,0.018)` + `backdrop-filter:blur(8px)` + 顶部高光线 + 22px 圆角
  - **html 必须加 `html { background: #0e0e28; }`**

- **横版小尺寸**（知乎/头条号 1080×608）：
  - 模板：`/opt/data/workspace/Projects/html-ppt-skill/templates/cover-templates/horizontal-1080x608.html`
  - 与大尺寸同风格，等比缩小字号和间距
  - padding: 14px 20px，圆角 20px，字号整体降一档

**同步发布**：5 个版本内容差异化足够，可以同一天同步发布，不会被判重复。

### 分发文件管理规范

做分发版本时，按以下规范组织文件：

**1. 文件夹结构**
为每篇文章建立独立文件夹，放在 `稿件库/已适配/` 下。**建分发版本时，原稿直接移入文件夹，不再在稿件库根目录保留**：
```
已适配/YYYY-MM-DD-文章简称/
├── [原稿]-文章标题.md          # 原稿，带[原稿]前缀（从稿件库根目录移入）
├── 微信公众号-标题.md          # 各分发版本，以平台发表标题命名
├── 什么值得买-标题.md
├── 知乎-标题.md
├── 小红书-标题.md
├── 头条号-标题.md
├── 头条号-标题.docx            # 头条号同步生成 .docx
├── 封面-公众号.png             # 1550×660
├── 封面-什么值得买.png          # 1550×660
├── 封面-知乎.png               # 1080×608
├── 封面-头条号.png              # 1080×608
└── 封面-小红书.png              # 1080×1440
```

**2. 封面自动生成**
分发时自动生成以下类别的封面（用 `xhs-wx-cover-template` skill + `html2png.py`）：
- 微信公众号封面（1550×660）
- 什么值得买封面（1550×660，与公众号共用）
- 知乎封面（1080×608）
- 头条号封面（1080×608，与知乎共用）
- 小红书封面（1080×1440）

**3. 头条号 .docx 同步生成**

**4. 发表后归档（2026-05 确认）**
用户告知文章已发表后：
1. 在 `稿件库/已适配/` 找到对应分发文件夹
2. **只保留原稿 `[原稿]xxx.md`**（删除所有5平台分发版、封面、docx等）
3. 原稿文件名前加日期前缀，直接移动到 `已发表/YYYY/` 目录（**不用套文件夹**）

**归档示例**：
```
# 旧格式（已废弃）
已发表/2026/2026-05-21-文章标题/
├── [原稿]xxx.md
├── 微信公众号-xxx.md
└── ...

# 新格式（2026-05 当前）
已发表/2026/
└── 2026-05-21-文章标题.md  ← 原稿直接放
```

## 数据驱动工作流（必读）

### 创作前必读数据分析报告

在进行任何自媒体相关创作工作前，**必须先读最近一次的数据分析报告（HTML 版本）**。具体包括：
- 选题建议
- 各平台标题建议
- 发文节奏建议
- 文章创作与修改
- 分发版本制作

报告路径：`数据分析/YYYY-MM-DD-五平台数据分析报告.html`（由 skill: design-shi-data-report 生成）

**读完报告后必须做的事：**
1. 确认各平台近期表现趋势（哪些话题/标题类型有效）
2. 检查是否有上次结论需要修正
3. 将数据结论应用到当前创作任务中（标题公式、平台选择、内容角度）

**不再生成 Markdown 版本**：数据分析报告只输出 HTML 格式。

当用户问"下一篇写什么"或"发什么"时，按以下步骤判断：

**第一步：建立上下文**
1. 回顾最近刚发表的文章（可在 `已发表/` 目录查看）
2. 查看文章结尾是否有"下篇预告"（系列文章的关键线索）
3. 确认用户是否有明确指定

**第二步：检查稿件库**
1. 列出 `稿件库/` 下所有草稿
2. 筛选已有5平台标题、可直接分发的文章（优先）
3. 标记还在writing状态的半成品

**第三步：多篇文章逻辑关系判断**
1. **系列文章优先**：如果已发文章结尾有预告，优先发后续篇
2. **同系列关联**：同一话题的硬件篇/策略篇、前篇/后续篇应连续发
3. **数据验证**：同一系列的文章，发后可以验证前文的判断（如"AI话题在头条零流量"）

**第四步：结合数据结论**
- 回顾最近一次数据周报或数据追踪中的关键结论
- 如：AI话题在头条号零流量、NAS话题在头条展现1万-8万、什么值得买"免费+攻略"标题效果最好等
- 用数据结论过滤不适合某些平台的选题

**第五步：选题评估**
按 design-shi_article skill 的 HKR 质检 + 五维评估判断哪个值得先发

**第六步：给出完整建议（含平台适配和后续规划）**
不是只回答"发哪篇"，而是提供：
- 推荐文章 + 理由（结合发布节奏、数据结论、时效性）
- 主攻平台建议（每个平台的标题）
- 后续安排（接下来2-3篇的发文节奏规划）

**完整建议示例**：
```
推荐发：FileBrowser文件管理器
理由：
1. 发布节奏：连续4篇AI话题，需要换NAS口味
2. 数据结论：NAS话题在头条号展现8.5万，是主力阵地
3. 时效性：已写6天，再放就凉了
主攻平台：头条号+什么值得买+知乎
后续：硬件篇接档，失忆+向量系列连发，中间插NAS缓冲
```

### 数据报告生成

生成数据分析报告时，使用 **skill: design-shi-data-report**。

**不再生成 Markdown 版本**：报告只输出 HTML 格式。

报告内容要求：
1. 本周各平台数据汇总
2. **与上次结论的对比验证**：上次的判断是否被本次数据支持？哪些结论需要修正？
3. 本周新增洞察
4. 下周发布建议（基于数据结论）

### 配图归档

制作分发版本时，**封面配图必须放在所属文章的分发文件夹下**，而不是统一放在 `配图/` 目录。这样用户在分发文件夹里就能看到文章+封面的完整组合。

文件夹结构：
```
稿件库/已适配/YYYY-MM-DD-文章简称/
├── [原稿]-文章标题.md
├── 微信公众号-标题.md
├── 小红书-标题.md
├── ...
├── 封面-公众号.png
├── 封面-小红书.png
└── ...
```
