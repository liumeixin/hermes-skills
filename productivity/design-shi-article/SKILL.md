## 第一步：选题判断

### HKR 质检（快筛）

一个快速判断选题质量的框架，帮助决定"值不值得写"：

- **H (Happy/Hook)**：有没有吸引力？标题和开头能不能让人好奇想点进来？
- **K (Knowledge)**：有没有信息增量？读者看完能不能学到点新东西？
- **R (Resonance)** ：能不能引发共鸣？看完会不会让人产生"对对对，我也这么觉得"？

**判断标准**：三个维度至少占两个，才值得写。如果只有一个维度有把握，可以和用户沟通调整方向。

## 风格融合引擎

写文章时的核心方法：**学爆款的结构，用自己的语气**。

- 学结构：开头怎么抓人、中间怎么展开、结尾怎么收
- 不学语气：不用别人的口头禅、句式习惯
- 核心原则：用爆款的骨架，说自己的话

### 五维选题评估（精筛）

HKR 快筛通过后，用五维评估决定"值不值得花时间写"。每条判断为「是 / 一般 / 否」：

| # | 维度 | 判断标准（家电/数码/3C适配） |
|---|------|---------------------------|
| 1 | **热度/时效性** | 是不是当下热点或有持续搜索量？新发布的产品/政策/事件？ |
| 2 | **反常识/认知差** | 有没有和大众直觉相反的点？"大家都以为A，其实B"？ |
| 3 | **理解门槛低** | 普通小白能不能一句话看懂它好在哪/差在哪？不需要背景知识？ |
| 4 | **有坑/风险** | 有没有隐藏缺点、不适用人群、踩坑点？读者看了能避雷？ |
| 5 | **同质化程度** | 同类内容是不是已经被讲烂了？有没有缝隙角度？ |

**判定规则：**
- ≥3 条「是」→ 值得写，给最佳切入点 + 3个备选标题
- 2 条「是」→ 勉强能写，需要强切入角度，展示风险点让用户决定
- ≤1 条「是」→ 不值得写，直接说理由，给唯一可能的角度（用户坚持则标注风险继续）

**输出格式（精筛结果）：**
```
📊 五维评估结果
热度：✅ / 一般 / ❌（一句话理由）
反常识：...
理解门槛：...
风险/坑：...
同质化：...

结论：值得写 / 勉强能写 / 不值得写
最佳切入点：xxx
备选标题：1. xxx  2. xxx  3. xxx
⚠️ 最容易写废的地方：xxx
```

### 五维选题评估（理性校验，参考 RedBook Workflow）

HKR 通过后，用五维框架做理性校验，判断"现在写是不是最佳时机"：

| 维度 | 问的是什么 | 判断标准 |
|------|-----------|----------|
| **热度** | 这个话题现在有人关注吗？ | 有搜索量/讨论量 → 是；完全没热度 → 否 |
| **反常识** | 有没有"大部分人想错了"的点？ | 有反转/反直觉的切入角度 → 是 |
| **理解门槛** | 目标读者能秒懂吗？ | 不需要前置知识就能理解核心观点 → 是 |
| **风险** | 有没有争议/得罪人的可能？ | 有 → 标记风险点，评估是否值得冒 |
| **同质化** | 已经被写烂了吗？ | 同话题高质量文章 <5篇 → 否（蓝海）；>10篇 → 是（需差异化） |

**判定规则**：
- ≥3 条"是" → **值得写**，输出：推荐切入点、备选角度、易写废点
- 2 条"是" → **勉强能写**，需要找到差异化角度
- ≤1 条"是" → **不值得写**，建议换选题或给出挽救方向（标明风险）

**注意**：五维评估是 HKR 的补充校验，不是替代。HKR 判断"选题好不好"，五维判断"现在写合不合适"。

### 热点时效分级（参考 OpenClaw）

选题时判断热点时效性：

| 等级 | 标识 | 时间范围 | 策略 |
|------|------|----------|------|
| 🔥 黄金热点 | HOT | 0-6小时 | 抢时效，事件报道、快速表态 |
| ⭐ 白银热点 | WARM | 6-24小时 | 差异化，多方观点、深度分析 |
| 📌 青铜热点 | COOL | 24-72小时 | 角度创新，深层解读、趋势预判 |
| 🌿 常青热点 | EVER | 72小时+ | 价值沉淀，知识沉淀、方法论 |

**选择原则**：
- ✅ 优先选择黄金热点（时效性最强）
- ⭐ 白银热点需要差异化角度
- 📌 青铜热点需要独特视角
- 🌿 常青热点适合专业垂直领域（适合设计虱定位）

## 第二步：明确人机协作边界

### AI 擅长做的（放心交给 AI）

- **找证据和佐证**：给观点后找支撑/反驳的论据
- **找类比和比喻**：用生动的比喻解释抽象概念
- **按确定的角度扩写**：核心角度和标题已确定，AI填充细节
- **补充背景知识**：格式塔心理学、原理等
- **梳理逻辑和结构建议**

### 必须人来的（不能交给 AI）

- **第一手真实经历**：自己的踩坑故事、真实体验
- **核心创意角度**：让文章立住的灵感，AI 给不了
- **情绪表达**：用体感记忆，不是知识性描述
- **个人判断**：敢下明确好恶

### 理想协作流程

```
人：提供素材 + 核心观点 + 个人经历
 ↓
AI：补充背景知识 + 找证据类比 + 按角度扩写
 ↓
人：二次改写（加入自己的声音、打破节奏）
 ↓
AI：按风格检查 → 输出修改建议
 ↓
人：终审和定稿
```

## 第三步：写作风格

### 文章原型

根据内容类型选择写法：

1. **调查实验型**：亲自下场做一件事，然后报道发现。写法重心在过程叙事和发现的层层递进
2. **产品体验型**：带着读者一起体验。写法重心在场景演示和真实感受
3. **现象解读型**：观察到一个现象，层层深入分析。写法重心在好奇→研究→升维
4. **工具分享型**：分享实用工具，用个人故事包裹。写法重心在故事铺垫→工具展示
5. **方法论分享型**：分享长期积累的经验。每一节要有可执行的行动建议，坦诚说明学习成本

### 文章节奏要点（编辑反馈总结）

1. **开头 30 秒定生死** — 读者看到"上下文污染"这类基础概念就会关掉，因为太多人写过了。开头直接亮结果或抛反常识观点，不要从"你有没有这种感觉"的铺垫切入
2. **问题→方案路径要短** — 从问题描述到解决方案不要超过 3 个章节。科普原理（LLM 工作原理、上下文窗口等）能砍就砍，除非读者不理解就无法接受你的方案
3. **每个章节问自己** — "读者看到这里会想往下读吗？"如果答案是"不会"，说明这里太冗长或太基础
4. **原理类内容压缩成一句话** — "上下文太杂了，它分不清你要干嘛"比三段 LLM 科普有效十倍
5. **效果对比放前面** — 分群前 vs 分群后的对比，比解释"为什么分群有效"更有说服力

### 核心原则

- **读者定位**：小白（beginners），完全不懂技术
- **场景优先，参数靠后**：读者关心"能干什么"，不关心"怎么干的"。产品/工具类文章，使用场景和案例占 60%+ 篇幅，规格参数压到最小。如果发现自己在写 Q&A 式的参数列表或代码块，砍掉，换成一个具体场景
- **有观点**：不做信息搬运工，每篇文章要有自己的观点和态度
- **口语化**：像在和读者聊天，亲切自然
- **有故事性**：用故事承上启下，但不要喧宾夺主
- **案例真实**：用真实具体的场景，越具体越好
- **结尾能量**：正向结尾，让读者有收获感
- **亲自下场**：不是隔空评论，是真的去做那件事

### 风格技巧

- **节奏感**：句子时长时短，用逗号制造口语停顿。一句话自成一段来制造重点
- **论述打破**：故意加入口语打断来破坏严谨性。"就是...就是...就是单纯的..."
- **知识输出方式**：知识是"聊着聊着顺手掏出来"的，不是"下面我来科普一下"
- **私人视角**：用"我也面临这个问题"来连接个人经历和公共议题
- **判断力**：敢下判断，有明确好恶。表达是"我被打动了"而不是居高临下的点评
- **对立面理解**：先承认对方的处境是合理的，再切入自己的不同视角
- **层层剥开**：现象→表面解释→更深的追问→核心洞察
- **英雄之旅**：普通人被召唤→经历考验→获得结果→回到日常

### 删除原则（重要！）

- 删的是**意思重复、没有骨干的废话**
- **不是该有的内容要保留**
- 能用图片说明的就可以少写文字（图片暂由用户配）

### 绝对禁区（最容易暴露 AI 味）

1. **禁用套话**："首先...其次...最后"、"综上所述"、"值得一提的是"
2. **禁用过度结构化**：不用 bullet point 罗列观点，不大量加粗
3. **禁用冒号破折号**：用逗号代替冒号":"，不用破折号"——"
4. **禁用 AI 高频词**：
   - "说白了"
   - "意味着什么？"
   - "本质上"
   - "换句话说"
   - "不可否认"
5. **严禁假设性例子**：要真实细节，不要编造
6. **禁止空泛工具名**：要说具体名字，如 Claude Code、OpenClaw、Obsidian
7. **禁止教科书开头**："在当今AI快速发展的时代"这种空话

### 推荐口语词组

**转场过渡**：坦率的讲、说真的、我是真的觉得、怎么说呢、其实吧、你想想看、回到xxx这块

**表达判断**：我有时候觉得、我一直觉得、不是说xxx不行而是说、我自己的感受是

**承认自嘲**：说实话我也不确定、我自己也踩过坑、这个事儿我也遇到过的

**情绪表达**：太爽了、我当时就愣住了、想想就觉得兴奋、太离谱了

**拉近距离**：很多朋友可能不知道、可能有小伙伴纳闷、你如果关注这个领域的话

## 第四步：格式规范

### 文件头部 - 标题

文章开头**不放任何标题内容**，直接从前言开始。各平台标题统一在文末发布建议里。

### 前言

**每次写文章前，必须先读取模板文件获取最新前言格式：**
`_templates/设计虱文章开头前的前言.md`

当前版本（2026-05-27）：
```markdown
---

`Hi，我是执着于持续分享数码、家电、NAS、AI、软件技巧相关知识，坚持创作有深度、高质量作品的博主 设计虱聊科技。期待您的关注。`

```

```markdown

```

直接用 `####` + 标题文字，不加 `|` 前缀、不加颜色 span（微信公众号用编辑器上色）。

### 尾注

**每次写文章前，必须先读取模板文件获取最新尾注格式：**
`_templates/设计虱文章结尾后的尾注.md`

### Tags

在尾注后面放上与文章相关的标签，例如：#hermes #openclaw #obsidian #nas #docker

### 发布建议（文末附带）

文章末尾附上发布建议，供用户参考：

```
---
📋 发布建议
- 最佳发布时间：[时间段，如"工作日 12:00-13:00 / 20:00-22:00"]
- 平台适配：
  - 微信公众号：标题xxx，适合深度阅读
  - 什么值得买：标题xxx，突出性价比/实用
  - 知乎：标题xxx，突出观点/分析
  - 小红书：精简版标题xxx，配图建议xxx
  - 微信问一问：标题xxx，问题式，有洞察
- 推荐话题标签：[各平台热门话题tag]
- 系列化建议：[下一篇可以写什么，形成系列]
- 配图建议：[需要几张图，每张图的内容描述]
```

**注意**：发布建议是参考，用户根据实际情况调整。平台标题参考记忆中的"自媒体标题规范"（同时提供5个平台标题建议）。

### 微信公众号摘要（必加）

**重要**：微信公众号版本的文章，必须在文末添加摘要（引用块格式），用于发布时填写摘要栏。

- 位置：发布建议上方，`---` 之后
- 格式：`> 摘要内容`（使用引用块 `>` 开头）
- 字数：≤100字
- 内容：基于文章核心观点，用 humanizer-zh 技能改写，去除 AI 味
- 范例：
  ```
  > AI 突然"失忆"，不是它变笨了。查完日志才发现是压缩摘要的坑——推理模型接不了这活。三步修复：压缩模型换 GLM-4.5-flash、阈值拉到 75%、保护 30 条消息。发个 /new 重建会话就行。
  ```

**注意**：摘要也要通过 humanizer-zh 技能检查，去除三段式、说教语气、宣传感等 AI 写作特征。

## 写作流程

### 0. 动笔前先确认：写哪篇？

用户说"写文章"时，先确认是哪个素材/哪篇稿子。如果涉及具体文件名，读一下确认内容是否匹配，避免把 A 素材写进了 B 文件。

### 1. 选题确认

- 读取 `topics.md` 确认选题
- 如果没有现成选题，问用户想写什么
- 用 HKR 质检判断选题质量

### 2. 素材准备

#### 读取风格档案

写之前先检查 `写作系统/风格档案.md` 是否存在：
- **存在**：读取并严格参照风格档案写作（人称、节奏、句式、emoji、结尾方式）
- **不存在**：按 skill 中的风格指南写，首次写完后建议用户触发风格提取

**核心原则**：风格与"最优写法"冲突时，优先保风格一致性。

#### 知识库检索

**动笔前先搜知识库**，路径：`/opt/data/workspace/设计虱-知识库/`

搜索方向：
- 话题关键词：概念、技术名词、工具名等
- 相关的"概念/实体/来源"页面

如果知识库有相关内容：
- 提取其中的观点、数据、案例作为素材
- 融合进文章里，不要重复造轮子
- 如果发现知识库内容与文章角度有冲突，尊重知识库的已有沉淀，考虑调整文章角度

#### 竞品文章拆解（六维框架，参考 RedBook Workflow）

写之前先搜同话题的热门文章，用六维框架系统拆解，提炼可复用的结构模式：

**搜索策略**：
```
平台优先级：
1. 知乎（深度观点）
2. 公众号（10w+传播）
3. 小红书（年轻视角）

搜索词：[话题] + "深度分析" / "评论" / "观点"
筛选：点赞>1000 / 阅读>10w / 转发>500
```

**六维拆解框架**（对每篇爆款参考文章逐项分析）：

| 维度 | 分析要点 | 提取目标 |
|------|----------|----------|
| **标题** | 句式结构、情绪关键词、悬念/数字/对比技巧 | 可复用的标题模式 |
| **开头钩子** | 前2行用了什么手法抓注意力？故事/提问/数据/反差？ | 钩子类型清单 |
| **关键词与标签** | 核心词、SEO词、长尾词、平台标签策略 | 关键词覆盖清单 |
| **内容结构** | 用流程图还原骨架：`[钩子] → [展开A] → [展开B] → [结尾CTA]` | 可移植的结构骨架 |
| **语言风格** | 人称、口语化程度、情绪、专业度、节奏、特征表达 | 风格参考（仅做对比，不模仿） |
| **互动设计** | 结尾引导话术、评论区预判、共鸣触发点 | 互动策略 |

**拆解后必做——结构骨架提取**：

分析完后，输出一篇爆款的"结构骨架"（去掉具体内容，只留框架）：
```
[钩子：具体场景切入]
  → [板块A：核心观点 + 案例佐证]
  → [板块B：深入分析 + 个人体验]
  → [板块C：对比/反转 + 更深洞察]
  → [结尾：升华 + 行动呼吁]
```

**分析后思考**：
- 已有内容有什么优点？我能借鉴什么？
- 有什么角度是没人写过或写得不好的？
- 我能不能做到"超越"？（角度更深、案例更新、表达更有力）

### 风格融合引擎（结构学爆款，语气做自己）

**核心理念**：爆款文章的结构骨架可以借鉴，但语气、人称、表达方式必须是自己的。"用你的结构，说你的话"。

**操作流程**：
1. 从爆款拆解中提取**结构骨架**（开头钩子类型 → 板块展开方式 → 结尾收束模式）
2. 确认自己的**声音特征**：人称视角、口语化程度、标志性表达、节奏习惯
3. 融合写作：用爆款的结构框架填充自己的内容和语气

**融合原则**：
- ✅ 学结构：开头怎么抓人、中间怎么展开、结尾怎么收
- ✅ 学节奏：长短句交替、段落密度、信息推进速度
- ✅ 学钩子：什么类型的开头最容易留住读者
- ❌ 不学语气：不用别人的口头禅、句式习惯、情绪表达方式
- ❌ 不学立场：别人的判断和观点只做参考，不照搬

**使用场景**：当找到明确的爆款参考文章时启用。如果没有爆款参考，跳过此步骤，直接按第三步写作风格写作。

### 3.5 文章拆分决策（写之前判断）

当一篇文章涵盖两个以上独立主题、或内容深度跨度大时，先判断是否应拆分：

**必须拆分的信号（任中一条即拆）：**
- **受众分裂**：前半段面向入门读者，后半段面向进阶用户（如"IMA协作"面向所有人 vs"向量化策略"面向已搭知识库的人）
- **深度不匹配**：一个主题需要场景化讲解，另一个需要技术细节分析
- **篇幅膨胀**：合写超过 8000 字，读者阅读疲劳
- **发布策略冲突**：一个主题适合蹭热度拉流量，另一个适合做深度沉淀

**拆分原则：**
- 每篇聚焦一个核心问题，有独立的标题和结论
- 前一篇文末预告下一篇（"下一篇我会聊聊xxx"）
- 后一篇开头引用前一篇（"上一篇讲了xxx，这篇聊点更细的"）
- 两篇间隔 1-2 天发布，先发流量型、再发深度型
- 拆分后各自独立选题、独立入库

**不需要拆分的情况：**
- 两个主题是"问题→解决方案"的递进关系（如"痛点→方案"）
- 篇幅在 5000-7000 字以内，阅读节奏流畅
- 受众基本一致，只是角度不同

### 推荐开头方式（避免"你有没有遇到过"的机械模式）

**核心原则**：开头 30 秒定生死，直接亮结果或抛反常识观点，不要从铺垫切入。

**多样化的开头模式**：

| # | 模式 | 适用场景 | 示例 |
|---|------|----------|------|
| 1 | **场景切入** | 日常生活场景，快速建立代入感 | "上周在朋友家，他问我：为什么你的NAS跑得比他的快？" |
| 2 | **反常识抛结论** | 先抛颠覆认知的观点 | "别再买贵的了——200块的路由器照样吊打千元旗舰" |
| 3 | **数据冲击** | 用具体数字抓住注意力 | "我花了3天测试了12款NAS系统，发现一个颠覆认知的事实..." |
| 4 | **故事开场** | 自己亲身经历，有画面感 | "凌晨三点，我被客服电话吵醒..." |
| 5 | **直接提问（慎用）** | 只有在问题足够痛时才用 | "你是不是也被路由器信号折磨过？" |
| 6 | **引用/热点** | 借助近期热点事件 | "昨天苹果发布会上的这个功能，其实半年前我们就用上了" |
| 7 | **否定式开场** | 先打破常见认知 | "大部分人都在犯同一个错误——买NAS只看硬盘位" |

**绝对避免的开头**：
- ❌ "你有没有遇到过..."
- ❌ "在当今AI快速发展的时代..."
- ❌ 任何宏大叙事的铺垫
- ❌ 从概念定义切入（"所谓XXX，是指..."）

### 背景铺垫

- 简要科普，让非专业读者也能看懂
- 方式是聊天式���，��是"下面我来讲讲"

### 核心内容

- 分几个板块展开
- 每个板块有：一个明确观点 + 至少一个具体场景/案例 + 扣主线的一句话

### 升华

- 从具体事件拉到更大的参照物
- 不是论文式总结，更像是"聊着聊着自然想到了"

### 收尾

- 引用收尾、哲思余韵、行动呼吁、信念宣言、回环呼应

## 错误处理

### 选题评估失败
```
HKR 质检不通过（只有1个维度有把握）
→ 与用户沟通调整方向
→ 尝试换角度重新评估
→ 如果仍然不通过，建议换选题
```

### 知识库无相关内容
```
搜索知识库未找到相关素材
→ 使用 web_search 补充背景知识
→ 标注信息来源
→ 继续写作流程
```

### 竞品文章找不到
```
同话题热门文章不足
→ 扩大搜索范围（相关话题）
→ 使用通用结构模板
→ 跳过竞品拆解，直接写作
```

### 格式检查失败
```
前言/尾注格式不符合规范
→ 读取模板文件重新生成
→ 替换不合规内容
→ 重新检查
```

### 五平台适配冲突
```
某平台不适合发布（如AI话题发头条）
→ 标注该平台不建议发布
→ 只适配适合的平台
→ 在发布建议中说明原因
```

## Anti-Patterns

**❌ 不要做的事：**
- 不要跳过 HKR 质检直接写作 → 每个选题必须通过质检
- 不要跳过风格档案读取 → 每次写作前必须确认风格一致性
- 不要用假设性例子 → 必须用真实细节，不要编造
- 不要用 AI 高频词 → 严格遵守"绝对禁区"列表
- 不要教科书式开头 → 避免"在当今..."、"你有没有遇到过..."
- 不要跳过 humanizer-zh 检查 → 每篇文章必须去AI味
- 不要自动创建五平台版本 → 等用户确认"分发"再做

## 注意事项

- 不要只给技术方案，要给背后的原因
- 用比喻、故事让内容更易懂
- 案例用真实场景，越具体越好
- 写完后自己读一遍，检查是否有片汤话
- 绝对禁区的东西要反复检查

### 质量自检清单（写完后检查）

写完后对照检查一遍：

| 维度 | 检查点 | 通过标准 |
|------|--------|----------|
| **内容质量** | 有观点吗？是搬运还是原创？ | 有独特观点，非搬运 |
| **情感共鸣** | 读者能"对对对"吗？有金句吗？ | 能戳中情绪，有记忆点 |
| **可读性** | 段落太长吗？节奏乱吗？ | 段落适中，长短句交错 |
| **传播性** | 标题吸引人吗？想转发吗？ | 标题有悬念，想转发 |
| **真实性** | 案例是真的吗？数据有来源吗？ | 真实案例，标注来源 |

**评分参考**（不必完全照搬，但心里要有数）：
- ⭐⭐⭐⭐⭐ 90-100分：优秀，直接发
- ⭐⭐⭐⭐ 80-89分：良好，微调可发
- ⭐⭐⭐ 70-79分：合格，建议优化
- ⭐⭐ 60-69分：需优化，重写核心部分
- ⭐ <60分：不合格，重新构思

## 五平台内容适配（写完一篇 → 适配五平台）

一篇完整文章写好后，适配5个平台发布。核心原则：**话题匹配度 > 内容质量**（数据报告显示同一篇内容在不同平台可差2-3个数量级）。

### 适配策略总览

| 平台 | 内容深度 | 标题风格 | 核心指标 | 典型大小 |
|------|----------|----------|----------|----------|
| 微信问一问 | 精简版（砍掉30-40%） | 问题式"怎么/为什么"，有洞察 | 收藏+回答质量 | 4-6K |
| 什么值得买 | 完整版 | 突出性价比/实用，加[标签]，≤30字 | 收藏 | 10K |
| 知乎 | 完整版 | 问题式"怎么/为什么"，有洞察 | 评论+收藏 | 10K |
| 微信公众号 | 精简版（砍掉30-40%） | 痛点式，20字内，有悬念 | 转发 | 4-6K |
| 小红书 | 大幅精简（砍掉60-70%） | emoji+感叹号，抓眼球，≤20字 | 封面点击率 | 2-3K |

### 各平台适配要点

**微信问一问**（问答互动主场，适合深度观点输出）
- 精简版：砍掉部署细节，保留核心场景和结论
- 标题用问题式（"怎么..."、"为什么..."），有洞察有观点
- 回答质量是核心指标，内容要有干货感
- AI 话题在问一问有一定关注度，可以发
- 适合做长尾搜索流量

**什么值得买**（收藏主场，有带货潜力）
- 完整版，标题加平台标签如 `[NAS必备]`、`[踩坑实录]`
- 收藏是最核心指标，内容要有"干货感"
- 知识库搭建类内容可挂商品链接测试
- 标题突出省钱/实用/具体工具名

**知乎**（长尾搜索主场）
- 完整版，标题改问题式（"怎么..."、"为什么..."）
- 大众话题有爆款潜力（5万+阅读），AI话题天花板约200-300
- 文章可持续获取搜索流量1-2年
- 生活场景切入 > 纯技术分析

**微信公众号**（AI话题主战场）
- 精简版：砍掉部署细节、进阶配置等技术内容，保留核心场景和结论
- 标题痛点式（"要SSH？...一步搞定"），20字内
- 转发率是核心传播杠杆（>5%算爆款）
- 叙事型/资讯型标题推荐权重低

**小红书**（互动主场，封面点击率生死线）
- 大幅精简：只留"是什么+怎么装+安全提醒"，砍掉多目录管理、权限控制等进阶内容
- 加emoji（💡🐳⚠️👇），段落更短，口语感更强
- 结尾加互动引导（"你们有没有类似好用的Docker工具？评论区聊聊👇"）
- 封面点击率>15%才能获得万级曝光，封面是第一优先级
- 技术术语必须翻译（ACL→"给AI设个门禁"）
- 有活动话题时优先适配（如 #榨干设备howto），标题加 `howto|` 前缀

### 适配操作流程

1. 确认完整版文章质量通过自检
2. 为5个平台拟定差异化标题（参考文章顶部的5平台标题区）
3. 创建 `稿件库/已适配/` 目录
4. 生成5个版本：
   - 什么值得买/知乎/问一问：从完整版复制，替换标题行
   - 微信公众号：精简版，砍掉技术细节，保留场景和结论
   - 小红书：大幅重写，加emoji，加互动引导，精简到2-3K
5. 文件命名：`YYYY-MM-DD-平台名-文章简称.md`
6. **用户编辑某一版本后，同步改动到其他版本**（用户通常会改一个版本，Agent负责把改动 propagate 到其余4个）
7. 跑 humanizer-zh 质量检查

**Docker 部署说明**：绿联 NAS 有 Docker Compose 管理器，文章里写 compose YAML 配置即可，用户粘贴到管理器直接部署，不用写 `docker run` 或 `docker compose up` 命令。

### 发布节奏

可同步发布（5个平台内容差异化足够，不判重复），也可分批：
- Day 1：什么值得买+知乎（完整版同步）
- Day 2：微信公众号+微信问一问（精简版/问答版）
- Day 3：小红书（差异化最大，需要单独做封面图）

## 批量审查已有稿件

当用户要求"过一遍稿件库"或"检查格式"时，按以下流程操作：

### 审查清单（逐篇检查）

| # | 检查项 | 通过标准 | 常见问题 |
|---|--------|----------|----------|
| 1 | **5 平台标题** | 文件头包含：微信公众号、什么值得买、知乎、小红书、微信问一问 | 早期稿件可能只有 3 个；"头条号"已替换为"微信问一问" |
| 2 | **标题顺序** | 固定顺序：微信公众号 → 什么值得买 → 知乎 → 小红书 → 微信问一问 | 顺序不一致，读者/编辑对照时容易混乱 |
| 3 | **前言格式** | `` `Hi，我是...设计虱聊科技。期待您的关注。` `` 首尾均有反引号，前后有 `---` | 旧版用"大家好啊...醉雪梅花"，需替换为新版 |
| 4 | **尾注格式** | 两段话均有反引号包裹；emoji 为 `ヾ(◍°∇°◍)ﾉﾞ` | 反引号经常只包了一半（缺开头或结尾） |
| 5 | **Tags** | 尾注前有 `#标签` 行 | 早期稿件可能缺 Tags |
| 6 | **发布建议** | 文末有 `📋 发布建议` 块，含最佳时间、5 平台适配、标签、系列化、配图 | 最容易缺的部分 |
| 7 | **小标题格式** | `#### 标题内容`（不带 `\|` 前缀） | 早期稿件可能有 `\|` 前缀，需清理 |

### 执行步骤

1. 读取稿件库所有 `.md` 文件
2. 对每篇按审查清单逐项检查，输出表格（✅/⚠️ + 缺什么）
3. 优先修前言/尾注反引号（批量 patch）
4. 补缺的平台标题（参考文章内容自行拟定，风格对齐已有标题）
5. 补发布建议（参考已有文章的发布建议格式，内容根据文章主题生成）
6. 用 Python 脚本批量验证所有文件是否通过

### 验证脚本模板

```python
checks = {
    "前言格式": "期待您的关注。`" in content,
    "尾注格式": "`坚持创作有深度" in content and content.count("`坚持创作有深度") == 1,
    "发布建议": "📋 发布建议" in content,
    "小标题格式": "#### " in content and "#### |" not in content,
}
```
